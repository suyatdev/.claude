### Layman summary

This spec builds the Tea Room's two request/response-shaped community rooms:
the Werk Room (a season-scoped feed of short posts with independent 💅/🔥
reaction toggles, no replies) and The Library (an admin-curated deck of hot
takes users vote sip/spit on exactly once, with the vote split hidden by the
server until the caller has voted). The third prototype room, Main Stage live
chat, is deliberately deferred — its transport is the exact decision stack-brief
D8 kept open — and appears only as a non-clickable teaser card in the hub.

The design is disciplined: four narrow tables in migration 0007, all counts
derived on read (the same stance ADRs 0007/0014 established, so no
counter-drift bugs), zero new dependencies, and a new fail-closed optional-auth
dependency (missing token → anonymous, invalid token → 401, never treated as
anonymous). Every boundary has an explicit error answer — 401/403/404/409/422
are each specified with client behavior for the 409 duplicate-vote case — and
the deferral list is honest and long (threads, moderation, admin UI, vote
changing, presence), each with the reason recorded in companion ADR 0016 as a
human-owned decision with rejected alternatives.

I verified the spec's load-bearing claims against the codebase and all held:
`get_current_user` exists in `api/deps.py` and the optional variant is
genuinely new; the `EventType` enum precedent (`Enum(EventType,
name="event_type")`) exists exactly as cited; the `withAuth()` public-page
pattern is live across the cast/seasons pages; `nav-rail.tsx:37` carries the
`soon` badge this spec removes; the latest migration is 0006 so 0007 is next;
`backend/scripts/seed_demo.py` exists. The spec lives at `docs/specs/`, the
canonical path `.claude/project-standards.md` §4 pins for this repo (project
layer overrides the global `docs/superpowers/specs/` default). Gherkin
scenarios cover good, bad, and edge cases (idempotent repeat reactions,
pagination at an exact multiple of 50, cross-season 404s, deck-finished and
empty states, split-hidden-until-vote for both anonymous and authed-unvoted
callers). No blocking violations found.

### Notes (non-blocking)

- **Toolchain granularity:** §9 quotes the backend constraint ranges verbatim
  from `backend/pyproject.toml` (`fastapi>=0.115,<1`, `sqlalchemy>=2.0,<3`,
  `alembic>=1.13,<2`; pytest is `>=8,<9`) and pins Next.js `16.2.10` / React
  `19.2.4` exactly; Vitest (`4.1.10` in `package.json`) is referenced via spec
  0005 §9 rather than restated. Acceptable per the spec-0006 round-1 precedent:
  zero new dependencies, exact pins live in `uv.lock`/`pnpm-lock.yaml`, and the
  pin-exact rule targets newly introduced deps. Flagged for transparency only.
- **Unknown `take_id` → 404:** §6 states the 404 rule explicitly for the
  reaction endpoints ("Unknown `post_id`, or post not in this season → 404")
  but not for `POST …/takes/{take_id}/votes` / admin `PATCH`. It is inferable
  from the season-scoped preamble and §10's "cross-season 404s" test
  requirement; one sentence would remove the inference.
- **Invalid `vote` value:** `TeaTakeVoteRequest` constrains `vote` to
  `"sip" | "spit"` by schema comment (→ 422 via validation), but no scenario
  enumerates the bad-value case the way "Unknown `kind` → 422" does for
  reactions. Implementer should add the 422 test alongside the schema.
- **No rate limiting on authed writes:** deliberate — ADR 0016 records the room
  as a trusted-beta space until moderation primitives (spec §12) exist. Fine
  for this slice; revisit before opening beyond the beta.
- **User-content rendering:** post bodies and take bodies are user/admin
  free text. The spec proposes nothing that bypasses React's default escaping;
  implementer must keep it that way (no `dangerouslySetInnerHTML` per
  writing-secure-code §2).

### Layman summary

Same design as round 1 — the two request/response-shaped Tea Room rooms, with
Main Stage live chat deferred per D8 — plus three targeted edits, all of which
tighten the spec rather than expand it. First, a new "Unknown-id rule,
uniformly" bullet in §6 gives the vote and admin-PATCH endpoints the same
explicit 404 contract the reaction endpoints already had (unknown id,
cross-season, or — for votes only — an inactive take), and pins invalid `vote`
values to 422; this closes round-1 notes 2 and 3, which had flagged exactly
these as inferable-but-unstated. The carve-out is coherent: votes on inactive
takes 404 (an invisible take shouldn't accept votes or leak its existence),
while admin PATCH can still reach an inactive take — necessary, since PATCH is
how a take gets reactivated. Second, a "Feed counts in one pass" bullet
requires reaction counts and viewer state to come from grouped queries over
the page's post ids rather than a per-post query loop — the same
derived-on-read-without-N+1 stance spec 0006 §5.1 set, a constraint on the
implementer, not a new feature. Third, §10's backend test list now names the
invalid-vote-422 and vote-on-inactive-take-404 cases, so the new contract
lines are test-backed rather than aspirational.

Nothing else changed (verified: §9 toolchain claims still match
`backend/pyproject.toml` and `frontend/package.json` exactly; zero new
dependencies). No edit introduced speculative scope, weakened an error
contract, or touched the security posture — fail-closed optional auth,
server-enforced split hiding, admin-gated curation, and the two-field-only
admin PATCH all stand as before. No blocking violations; verdict is a clean
pass.
