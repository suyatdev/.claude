# Compliance verdict — spec 0006 The Cast (Roster & Queen Profiles)

Spec: `docs/specs/0006-the-cast.md` (companion ADR `docs/decisions/0014-cast-derived-stats.md`)
Slug: `0006-the-cast`

## Round 1 — 2026-07-18

**Verdict: PASS** (confidence: high)

### Layman summary

This spec turns the disabled "The Cast" nav item into a live, public per-season
roster grid plus an individual queen profile page, styled to the Dragquette
prototype. It is a pure read slice: no database migration, no new dependencies.
The two stats it surfaces — "pick %" (share of the latest-locked round's
brackets that put a queen at rank 1) and elimination status
(active / eliminated E# / winner) — are derived on read from data that already
exists (locked brackets + the outcome ledger). It correctly reuses the scoring
service's supersession resolution instead of forking it, and it is honest about
what it cannot do yet: per-episode track record and wins count are explicitly
deferred because the ledger stores no placement-level data (ADR 0007).

I verified every load-bearing claim against the codebase and all five held:
`Queen` already has `name`+`tagline`; `scoring._authoritative_events` exists and
handles supersession; scoring never touches `RETURN` (so status mapping is
genuinely new, not a silent duplication); the public-route pattern matches the
merged leaderboard page; and `Bracket.predictions` is a JSONB list with
`predictions[0]` = rank-1 slot. The spec lives at `docs/specs/`, which the
repo's `.claude/project-standards.md` §4 pins as the canonical location
(overriding the global default path).

The spec is strong on the writing-specs rubric: Gherkin scenarios cover good,
bad, and edge cases (pre-lock null, `D==0` null, the 0%-vs-null distinction,
superseded correction, returned queen, winner, unknown season/queen, signed-out
render); the API contract is a concrete YAML schema with status codes; error
handling is stated at every boundary; and it is aggressively YAGNI (defers
everything that lacks backing data). No blocking violations found.

### Notes (non-blocking)

- **Toolchain granularity:** §8 names existing deps (FastAPI, SQLAlchemy 2,
  pytest, Next.js 16, React 19, Vitest) at major-version granularity rather than
  exact pins. Acceptable here — the spec introduces zero new dependencies and
  reuses versions already pinned in the repo's lockfiles; the pin-exact-version
  rule targets newly introduced deps, of which there are none. Flagged only for
  transparency.
- **Shared-symbol change:** §4 offers "drop the underscore or re-export" to make
  `_authoritative_events` reusable. Prefer the re-export path — dropping the
  underscore renames a symbol scoring.py already calls internally
  (scoring.py:119), whereas re-export honours the extend-not-alter invariant for
  shared modules.
- **Rounding mode:** §5.1 uses `round(100 * count / D)`. Python's `round` is
  banker's rounding, and per-queen percentages need not sum to 100. Deterministic
  and well-defined, but worth an implementer comment so the choice is explicit.
